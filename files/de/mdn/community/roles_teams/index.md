Der Erfolg und das Wachstum des MDN Web Docs Projekts beruhen zu einem großen Teil auf unserer Community von Mitwirkenden. Einige Mitwirkende haben einen Teil ihrer Zeit zugesagt, um bei den täglichen Aufgaben zur Führung von MDN Web Docs zu helfen. Änderungen an der Seite, einschließlich Wartungsaufgaben, werden von Mitarbeitern, Auftragnehmern und einem Netzwerk von Partnern durchgeführt, die sich alle der Gesundheit, dem Wachstum und der Pflege von MDN Web Docs widmen. Das Projekt stützt sich stark auf [Rollen](#rollen) und [Teams](#teams) in der [MDN-Organisation auf GitHub](https://github.com/mdn), um Änderungen aus diesen verschiedenen Gruppen zu verwalten und zu integrieren. Eine Liste der Mitglieder der Organisation kann [hier gefunden werden](https://github.com/orgs/mdn/people).

Community-Beiträge helfen diesem Open-Source-Projekt enorm.
Mitwirkende können ihre Arbeit an den MDN Web Docs nutzen, um ihre Schreib-, Technik- und Kollaborationsfähigkeiten zu zeigen sowie die Fähigkeit, mit Menschen aus unterschiedlichen Hintergründen zu arbeiten. Dieser Abschnitt beschreibt die Rollen, die Sie während Ihrer ehrenamtlichen Tätigkeit im MDN Web Docs Projekt übernehmen können.

## Rollen

Im MDN Web Docs Projekt können Sie die Rolle eines [Mitwirkenden](#mitwirkender), eines [Organisationsmitglieds](#organisationsmitglied), eines [Maintainers](#maintainer) oder eines [Eigentümers](#eigentümer) übernehmen. Der Fortschritt von einer Rolle zur nächsten ist ein Schritt-für-Schritt-Weg.
Mit dem Fortschritt Ihrer Verantwortung könnten Sie mehr als eine Rolle gleichzeitig ausüben.
Rollen wie [eingeladener Experte](#eingeladener_experte) können direkt übernommen werden, wenn Sie Fachwissen in einem bestimmten Bereich nachgewiesen haben.

Unabhängig von der Rolle, die Sie in diesem Projekt übernehmen, sind Sie immer ein [Mitwirkender](#mitwirkender).
Ein **Mitwirkender** ist die Basisrolle und alle anderen Rollen bauen darauf auf.
Wenn Sie also in irgendeiner Funktion an diesem Projekt arbeiten, müssen Sie die Anforderungen der Rolle des Mitwirkenden erfüllen.

### Mitwirkender

Mitwirkende, oder _Community-Teilnehmer_, tragen mit ihrer Zeit, ihren Fähigkeiten, Meinungen und Ideen zum Projekt bei.
Mitwirkende arbeiten direkt an dem Projekt und fügen ihm Wert hinzu.
Neben dem Schreiben und Testen von Code umfassen die Beiträge auch das Erstellen und Aktualisieren von Dokumentationen, Recherchen, Beheben von Fehlern und das Helfen anderer Community-Mitglieder.

Je nach Häufigkeit Ihrer Beiträge können Sie jemand sein, der gelegentlich beiträgt oder ein aktiver Mitwirkender.
Wenn Sie einen großen Einfluss auf das Projekt nachweisen, können Sie als [Spotlight-Mitwirkender](#spotlight-mitwirkender) nominiert oder zum [Organisationsmitglied](#organisationsmitglied) befördert werden.

Wenn Sie neu hier sind und ein Mitwirkender werden möchten, lesen Sie bitte unseren [Leitfaden für Beiträge](https://github.com/mdn/content/blob/main/CONTRIBUTING.md) und die [Repositorien in der MDN GitHub Organisation](https://github.com/orgs/mdn/repositories).

Als Mitwirkender können Sie sich an dem Projekt beteiligen, indem Sie sich an folgenden Aktivitäten beteiligen:

- Teilnahme an Community-Diskussionen auf den [Kommunikationskanälen](/de/docs/MDN/Community/Communication_channels).
- Anderen Mitwirkenden bei ihren Pull Requests und Problemen helfen oder neue Mitwirkende betreuen.
- Fehlermeldungen einreichen. Schauen Sie sich die [Art der Probleme an, die Sie öffnen können](https://github.com/mdn/content/issues/new/choose) im MDN-Repository [`content`](https://github.com/mdn/content).
  Wenn Sie einen Plattformfehler bemerken, können Sie [ein Problem melden](https://github.com/mdn/yari/issues/new/choose) im MDN-Repository `yari`.
- Kommentare zu Problemen abgeben, um Gespräche zu einer fruchtbaren Lösung zu führen.
- Offene Probleme angehen (zum Beispiel im [`content`](https://github.com/mdn/content/issues) Repository) durch das Einreichen von [Pull Requests](/de/docs/MDN/Community/Pull_requests).
- Teilnahme an Community-Events.
- Mithilfe zur Förderung des MDN-Projekts.

**Anforderungen:**

Um ein Mitwirkender zu sein, müssen Sie folgendes befolgen:

- [Mozillas Verhaltenskodex](https://www.mozilla.org/en-US/about/governance/policies/participation/)
- Richtlinien für Beiträge (prüfen Sie die Datei `CONTRIBUTING.md` in jedem Repository; zum Beispiel diese sind die [Richtlinien für Beiträge](https://github.com/mdn/content/blob/main/CONTRIBUTING.md) für das `mdn/content` Repository).

**Privilegien:**

Mitwirkende genießen die folgenden Privilegien:

- Einladungen zu Mitwirkenden-Events.
- Berechtigung, ein [Organisationsmitglied](#organisationsmitglied) zu werden.

### Organisationsmitglied

[Organisationsmitglieder](https://github.com/orgs/mdn/people) sind etablierte [Mitwirkende](#mitwirkender), die regelmäßig am MDN Web Docs Projekt teilnehmen und dazu beitragen.
Von ihnen wird erwartet, dass sie im Interesse des Projekts handeln.

**Anforderungen:**

Um ein Organisationsmitglied zu sein, müssen Sie eine oder mehrere der folgenden Anforderungen erfüllen:

- Zwei oder mehr Pull Requests eröffnet, die zusammengeführt wurden und zwei oder mehr Probleme lösen.
- Seit mindestens zwei Monaten zu MDN Web Docs Projekten beigetragen.
- Aktiv zu mindestens einem Projektbereich beigetragen.

Die folgenden zwei Anforderungen sind obligatorisch:

- Aktivierte [Zwei-Faktor-Authentifizierung](https://docs.github.com/en/authentication/securing-your-account-with-two-factor-authentication-2fa/configuring-two-factor-authentication) für Ihr GitHub-Konto.
- Aktivierte [signierte Commits](https://docs.github.com/en/authentication/managing-commit-signature-verification/signing-commits).

**Privilegien:**

Organisationsmitglieder haben Privilegien auf der [Organisationsebene](https://github.com/mdn) auf GitHub.

### Maintainer

Maintainer sind etablierte [Mitwirkende](#mitwirkender), die für ein oder mehrere MDN-Projekte verantwortlich sind.
Von ihnen wird erwartet, dass sie an Entscheidungen über die Richtlinien und Prioritäten des Projekts teilnehmen. Sehen Sie sich den [Prozess](#nominierung_eines_maintainers) an, um jemanden als Maintainer zu nominieren.

Als Maintainer beteiligen Sie sich an folgenden Aktivitäten:

- Festlegung von Prioritäten für das Projekt, für das Sie verantwortlich sind.
- Teilnahme an Community-Treffen.
- Betreuung neuer und bestehender Mitwirkender in allen anderen Rollen.
- Auf Grundlage des Fähigkeitsprofils Vorschläge, Genehmigungen oder Implementierungen in Ihrem Projektbereich:
  - Verbesserungen des Codes und der Infrastruktur
  - Verbesserungen des Inhalts
  - Verbesserungen der Prozesse

**Anforderungen:**

Um als Maintainer in Frage zu kommen, müssen Sie eine oder mehrere der folgenden Anforderungen erfüllen:

- Mindestens sechs Monate Erfahrung als [eingeladener Experte](#eingeladener_experte) erworben.
- Breites Wissen über das Projekt in mehreren Bereichen gezeigt.
- Die Fähigkeit bewiesen, Urteilsvermögen zum Wohl des Projekts unabhängig von den Einflüssen anderer Mitglieder auszuüben.
- Die Qualität der Betreuung anderer Mitwirkender gezeigt.
- Zugestimmt, mindestens 16 Stunden pro Monat an dem Projekt zu arbeiten.
- Teilnahme an dem Community-Meeting, das alle zwei Monate stattfindet.

> [!NOTE]
> Wenn Sie denken, dass jemand für diese Rolle geeignet ist, können Sie [einen Maintainer nominieren](#nominierung_eines_maintainers).

**Privilegien:**

Maintainer haben die Berechtigungen, Pull Requests zu genehmigen und zusammenzuführen.

### Eigentümer

Eigentümer haben umfassende Berechtigungen, um Benutzer und [GitHub-Teams](https://github.com/orgs/mdn/teams) zu verwalten, den Zugriff über Repositories in der [MDN-Organisation](https://github.com/mdn) zu pflegen, Repository-Einstellungen zu verwalten und in die Produktion zu deployen.
Eigentümer sind an alle Anforderungen anderer Mitwirkenderrollen gebunden.

> [!NOTE]
> Die Rolle eines Eigentümers ist derzeit auf Mozilla-Mitarbeiter beschränkt.

**Anforderungen:**

Zusätzlich zu den Verantwortlichkeiten anderer Mitwirkenderrollen haben Eigentümer die folgenden Verantwortlichkeiten:

- Beachtung und Durchsetzung der MDN-Team-Normen, einschließlich der [Community-Teilnahmerichtlinien](https://www.mozilla.org/en-US/about/governance/policies/participation/) und [Mozilla-Richtlinien](https://www.mozilla.org/en-US/about/governance/policies/).
- Befolgung der MDN-Organisationsrichtlinien und Führung durch Beispiel.
- Vorschlagen, Dokumentieren und Implementieren neuer Richtlinien über den [Pull-Request-Prozess](/de/docs/MDN/Community/Pull_requests).
- Verfolgung und Beitrag zu Themen und Diskussionen in der MDN-Organisation.
- Sicherstellen, dass ein Thema oder ein Pull Request innerhalb einer Woche von einem oder mehreren Mitgliedern Rückmeldung erhält.
- [Archivierung](https://docs.github.com/en/repositories/archiving-a-github-repository/archiving-repositories) oder Löschung nicht gepflegter Repositories.
- Diskussion über GitHub-Features, Auswahl der zu verwendenden Features und Dokumentation der Entscheidungen.

**Privilegien:**

Eigentümer können:

- Bei Bedarf Organisationsbesitzer und Mitglieder hinzufügen und entfernen.
- Bei Bedarf Mitarbeiter zu spezifischen Repositories hinzufügen und entfernen.
- Bei Bedarf Repositories hinzufügen (als neue Projekte oder Transfers).

## Besondere Rollen

Einige Mitwirkenderrollen haben detailliertere Verantwortlichkeiten und besondere Berechtigungsvoraussetzungen. Dazu gehören [Spotlight-Mitwirkender](#spotlight-mitwirkender), [eingeladener Experte](#eingeladener_experte) und [Community-Manager](#community-manager).

### Spotlight-Mitwirkender

Spotlight-Mitwirkende sind Personen, die über ihre Beiträge zu den MDN Web Docs hinausgehen.
Ihre Beiträge sind in Form von Pull Requests zur Verbesserung des Projekts, Hilfe für Community-Mitglieder auf verschiedenen [Kommunikationskanälen](/de/docs/MDN/Community/Communication_channels) oder Lernforen oder Bereitstellung von Feedback zu GitHub-Issues und Pull Requests.

Wir präsentieren einmal im Monat einen Spotlight-Mitwirkenden auf der [MDN-Website](/en-US/).
Sehen Sie sich den [Prozess](#nominierung_eines_spotlight-mitwirkenden) an, um jemanden als Spotlight-Mitwirkenden zu nominieren.

### Eingeladener Experte

Eingeladene Experten haben auf MDN eine nachgewiesene Erfolgsbilanz bei ihren Beiträgen, Teilnahme an Diskussionen und Rezensionen oder nachgewiesene Kenntnisse in einem bestimmten Fachgebiet.
Eingeladene Experten sind für ein spezifisches Themengebiet oder eine Komponente des MDN-Projekts verantwortlich.
Sie sind verantwortlich für die Überprüfung und Genehmigung von Pull Requests in ihrem Themen- oder Projektbereich, Beantwortung technischer Fragen und Pflege der allgemeinen Gesundheit ihres speziellen Projekts. Sehen Sie sich den [Prozess](#nominierung_eines_eingeladenen_experten) an, um jemanden als eingeladenen Experten zu nominieren.

Zusätzlich zu den Verantwortlichkeiten eines [Organisationsmitglieds](#organisationsmitglied) sind eingeladene Experten verantwortlich für:

- Befolgung des [Überprüfungsleitfadens](https://github.com/mdn/content/blob/main/REVIEWING.md).
- Überprüfung von Pull Requests in ihrem Themenbereich.
- Unterstützung anderer Mitwirkender, Reviewer zu werden.

Eingeladene Experten werden automatisch für die Überprüfung zugewiesen, wenn Pull Requests in ihrem Themenbereich geöffnet werden.
Wenn es mehr als einen Experten in einem Themenbereich gibt, werden sie nach einer [Lastverteilungsmethode](https://docs.github.com/en/organizations/organizing-members-into-teams/managing-code-review-settings-for-your-team#about-auto-assignment) Pull Requests zugewiesen.

**Anforderungen:**

Um als eingeladener Experte in Frage zu kommen, müssen Sie eine oder mehrere der folgenden Anforderungen erfüllen:

- Nachgewiesenes tiefgehendes Wissen in einem bestimmten Themenbereich.
- Verpflichtung, für ihren zugewiesenen Themenbereich verantwortlich zu sein.
- Unterstützung neuer und gelegentlicher Mitwirkender und Hilfe bei der Vorbereitung von Pull Requests für die Zusammenführung.
- Teilnahme an dem Community-Meeting, das alle zwei Monate stattfindet.

**Privilegien:**

Eingeladene Experten werden zum [Team der eingeladenen Experten](https://github.com/orgs/mdn/teams/invited-experts-and-co-maintainers) und zum entsprechenden Themen- oder Projektteam hinzugefügt. Eingeladene Experten können:

- Zugriff auf das erforderliche Repository für Commits und Genehmigungen sowie Zusammenführungen von Pull Requests.
- Andere Mitglieder zur Ernennung als eingeladene Experten empfehlen und darüber abstimmen.
- Wöchentliche MDN Web Docs Redaktionsbesprechungen besuchen.

### Community-Manager

Community-Manager haben in vielerlei Hinsicht eine besondere Rolle.
Community-Manager teilen viele der gleichen Verantwortlichkeiten wie ein [Maintainer](#maintainer).
Darüber hinaus haben Community-Manager folgende Verantwortlichkeiten:

- Bearbeitung von Berichten über Verstöße gegen unseren [Verhaltenskodex](https://github.com/mdn/mdn-community/blob/main/CODE_OF_CONDUCT.md), der [Mozillas Community-Teilnahmerichtlinien](https://www.mozilla.org/en-US/about/governance/policies/participation/) ist, und Entscheidung über die geeignete Vorgehensweise.
- Organisation und Durchführung von Community-Events.
- Organisation von Community-bezogenen Projektbesprechungen.
- Bestimmung von Medienstrategien zur Förderung des MDN-Projekts.
- Definition und Umsetzung der Erfahrung zur Einführung neuer Mitwirkender.
- Einführung neuer Mitwirkender und Benutzer.
- Sicherstellung der Gesundheit und des Wohlbefindens des MDN-Projekts und aller Teilnehmer.
- Identifikation und Unterstützung bei der Implementierung von Automatisierung zur Verbesserung der Projekt-Nachhaltigkeit.
- Treffen und Sicherstellen einer gesunden Beziehung zu Mitwirkenden und Partnern.
- Unterstützung bei der Triagierung von Problemen und der Überprüfung von Pull Requests, wo angemessen.
- Überwachung aller [Kommunikationskanäle](/de/docs/MDN/Community/Communication_channels).
- Hervorheben von Mitwirkenden, die außergewöhnliche Arbeit geleistet oder Engagement für das MDN-Projekt gezeigt haben.

### Nominierung eines Maintainers

1. Auf der Registerkarte `Issues` im `mdn/mdn` Repository klicken Sie auf die Schaltfläche [**New issue**](https://github.com/mdn/mdn/issues/new/choose) auf der rechten Seite.
2. Unter 'Nominate a maintainer' klicken Sie auf die Schaltfläche **Get started**.
3. Füllen Sie das Formular mit Details zu den Beiträgen der Person, die Sie nominieren, aus und senden Sie das Formular ab.

### Rücktritt oder Bewerbung um Emeritus-Status

Das Leben passiert und Ihr Engagement als Mitwirkender kann sich im Laufe der Zeit ändern. Je nach Ihrer Situation möchten Sie vielleicht:

- Eine Pause vom Projekt einlegen.
- Auf eine weniger anspruchsvolle Rolle umsteigen.
- Sich komplett von dem Projekt zurückziehen (für einen Emeritus-Status bewerben).

In all diesen Situationen können Sie Ihre Situation und Ihre aktuellen Engagement-Levels gerne mit dem [MDN-Team](#kontakt_mit_dem_mdn-team_aufnehmen) besprechen.

### Degradieren oder Entfernen inaktiver Mitwirkender

Ein Mitwirkender kann degradiert oder als Mitwirkender entfernt werden, wenn Verantwortlichkeiten und Anforderungen nicht erfüllt werden, einschließlich wiederholter Muster von Inaktivität oder eines Verstoßes gegen den [Verhaltenskodex](https://github.com/mdn/mdn-community/blob/main/CODE_OF_CONDUCT.md).

Eine Degradierung oder Entfernung eines Mitwirkers wird von einem Teilnehmer während eines Maintainer-Meetings vorgeschlagen.
Der Teilnehmer liefert unterstützende Informationen für den Degradierungs- oder Entfernungsvorschlag.
Nach der Diskussion stimmen Maintainer und Community-Manager ab, um die Entscheidung zu treffen.

Das Entfernen inaktiver Mitwirkender schützt das Projekt und seine Ergebnisse und eröffnet auch neuen Mitwirkenden die Möglichkeit, zu intervenieren.

Wir definieren Inaktivität als:

- Keine Beiträge zum Projekt für mindestens sechs Monate.
- Keine Antwort auf Kommunikation für mindestens drei Monate.

Inaktivität schadet dem Projekt; sie kann zu unerwarteten Verzögerungen, Schwund von Mitwirkenden und einem Vertrauensverlust in das Projekt führen. Mitwirkende müssen aktiv sein, um ein Beispiel zu setzen und Engagement für das Projekt zu zeigen.

Bitte kommunizieren Sie mit dem Community-Team, um eine Degradierung oder Entfernung zu vermeiden, wenn sich Ihre zeitlichen Verpflichtungen ändern; stattdessen können Sie proaktiv wählen, [für eine Weile zurückzutreten oder in einen Emeritus-Status zu wechseln](#rücktritt_oder_bewerbung_um_emeritus-status).

## Teams

Wir verwalten Teams mit der [GitHub Teams](https://docs.github.com/en/organizations/organizing-members-into-teams/about-teams) Funktion. Wenn Sie in ein Team aufgenommen werden, bedeutet dies, dass Sie Ihr Interesse bekundet haben, etwas enger mit dem Projekt verbunden zu sein. Das bedeutet auch, dass Sie einige zusätzliche Verantwortlichkeiten und Rechte haben, wie unten erklärt:

- Eine Person in einem Team wird häufig zur [CODEOWNERS](https://github.com/mdn/content/blob/main/.github/CODEOWNERS) Datei für ihre jeweiligen Themenbereiche von Interesse hinzugefügt.

- Wenn ein Pull Request Dateien in Ihrem Verantwortungsbereich berührt, basierend auf dem CODEOWNERS-Dokument, werden Sie automatisch als Reviewer zu einem Pull Request hinzugefügt, indem GitHubs [Lastverteilungsalgorithmus](https://docs.github.com/en/organizations/organizing-members-into-teams/managing-code-review-settings-for-your-team#routing-algorithms) verwendet wird.

- Mitglieder eines Teams haben einen höheren Repository-Zugriff. Repository-Berechtigungen werden nur denen zugewiesen, die Zugriff benötigen.

Die Teams in der [MDN GitHub Organisation](https://github.com/orgs/mdn/teams) umfassen:

- `@Core`: Core MDN Web Docs Team
- `@mdn-community-engagement`: Personen, die für Community-Engagement in unseren Repositories verantwortlich sind
- `@mdn-product`: Personen, die für das MDN Plus Produkt verantwortlich sind
- `@localization-team-leads`: Personen, die unsere individuellen Lokalisierungsteams leiten
- `@OWD`: Mitwirkende der gemeinnützigen Organisation Open Web Docs
- `@sre`: Site-Reliability-Ingenieure, die die MDN Web Docs unterstützen
- `@yari-content`: Das Dachteam für alle MDN Web Docs Content-Reviewer

  - Es gibt ein Unterteam für die verschiedenen Themenbereiche — Barrierefreiheit, Add-ons, CSS, HTML, HTTP, JavaScript, SVG, Web API und WebAssembly. Zum Beispiel gibt es `@yari-content-css` und `@yari-content-svg`.
  - Es gibt auch Unterteams für verschiedene Sprachen — brasilianisches Portugiesisch, Chinesisch, Französisch, Japanisch, Koreanisch, Russisch und Spanisch. Zum Beispiel gibt es `@yari-content-fr` und `@yari-content-ko`.

Um ein Mitglied eines Teams zu werden, müssen Sie:

- Zustimmen, unsere [Richtlinien zur Teilnahme an der Community](https://www.mozilla.org/en-US/about/governance/policies/participation/) einzuhalten.
- Zustimmen zu Mozillas [Anforderungen an den Commit-Zugriff](https://www.mozilla.org/en-US/about/governance/policies/commit/requirements/).
- Die [Zwei-Faktor-Authentifizierung](https://docs.github.com/en/authentication/securing-your-account-with-two-factor-authentication-2fa/configuring-two-factor-authentication) (2FA) in Ihrem GitHub-Konto einrichten.
